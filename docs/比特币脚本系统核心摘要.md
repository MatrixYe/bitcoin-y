## 指令=操作码+数据
Bitcoin Script 里的字节不是“一字节 = 一个普通操作码”这么简单。
例如 P2PKH 里的 script_pubkey 是：
```text
OP_DUP OP_HASH160 <20-byte pubkey hash> OP_EQUALVERIFY OP_CHECKSIG


```

它的真实字节大概是：

```text
76 a9 14 <20 bytes> 88 ac

```
这里的：
```text
0x76 = OP_DUP

0xa9 = OP_HASH160

0x14 = 不是 OP_14，而是 direct push：表示后面有 20 字节数据

<20 bytes> = 真正要压栈的数据

0x88 = OP_EQUALVERIFY

0xac = OP_CHECKSIG
```

所以 0x14 在这里不是“小整数 20”，而是“接下来读取 20 字节并压栈”。如果 parser 只输出 opcode: u8，解释器就还得重新理解这些字节边界，容易把“脚本解析”和“脚本执行”混在一起。

## 数据编码
0x01..=0x4b
直接表示 1 到 75 字节数据长度

OP_PUSHDATA1 = 0x4c
后面 1 字节表示长度，最多 255 字节

OP_PUSHDATA2 = 0x4d
后面 2 字节表示长度，最多 65535 字节

OP_PUSHDATA4 = 0x4e
后面 4 字节表示长度，理论上可表达很大长度

但实际执行时还有限制。v0.3.19 的 EvalScript 会检查：
```cpp
if (vchPushValue.size() > 520)
    return false;

```
也就是单个压栈元素最大 520 字节


### direct push 
direct push 不是具名 opcode，但它仍然是 Bitcoin Script 字节协议的一部分。Bitcoin Script 的规则是：
> 如果第一个字节在 0x01..=0x4b 之间，
> 这个字节本身不表示“操作码名字”，而是表示“后面有多少字节数据要压栈”。
例如：
```
03 aa bb cc
```
解释方式是：
```
0x03 = direct push，后面读取 3 字节
aa bb cc = 要压栈的数据
```
执行结果:
```
stack.push([aa, bb, cc])

```
v0.3.19 里也是这么做的。在 CScript::GetOp2，它先读一个字节
unsigned int opcode = *pc++;
然后在 script.h:527 判断：
```cpp
if (opcode <= OP_PUSHDATA4)
if (opcode < OP_PUSHDATA1)
nSize = opcode;
```

这句话就是 direct push 的核心：如果字节小于 OP_PUSHDATA1，也就是小于 0x4c，那这个字节直接表示数据长度。

能不能支持任意长度数据？ 不能。
Bitcoin Script 的 push 数据有几种编码方式：

### 编码策略
**同一段数据，有时可以用不同方式编码进脚本。**
假设要把 3 个字节压栈：

aa bb cc

最短编码是 direct push：

03 aa bb cc

含义是：

0x03 表示后面有 3 字节数据
aa bb cc 是数据

但也可以用 OP_PUSHDATA1 编码：

4c 03 aa bb cc

含义是：

0x4c = OP_PUSHDATA1
0x03 = 后面有 3 字节数据
aa bb cc 是数据

这两个脚本执行效果一样，都会把 aa bb cc 压栈。但字节流不同。

好的方面，这似乎提供了脚本压缩的可能。坏的方面，这会造成所谓的交易拓展性攻击。

### Direct、PushData1、PushData2、PushData4

Direct push 的长度范围只有75字节，如果push的数据超过这个数目就不够了。
所以使用PushData1、PushData2、PushData4来表示，下面n个字节表示接下来要push的数据的长度

例如压入 80 字节，80 = 0x50，而 0x50 已经是 OP_RESERVED了。所以要写成

```text
4c 50 <80 bytes>

```
表示:

```text
4c = OP_PUSHDATA1
50 = 后面 80 字节
<80 bytes> = 数据

```
